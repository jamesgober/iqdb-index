# iqdb-index v0.2.0 — The Index Trait

**The contract lands.** v0.2.0 turns the v0.1.0 scaffold into the load-bearing surface every iQDB index speaks: the object-safe `IndexCore` operational trait, the typed-construction `Index` trait, the `IndexStats` introspection struct, and the default batch shims — each documented with a runnable example and validated by a property-test suite against a brute-force reference index. This is the hard part of the roadmap, and it is not deferred.

## What is iqdb-index?

The polymorphism layer of the iQDB vector database. The engine calls `index.search(query, params)` and the right implementation runs, so iQDB supports plug-and-play index strategies (flat, HNSW, IVF, …) instead of being hardcoded to one. It is interface code — two traits plus one struct — and depends only on `iqdb-types`.

```rust
use std::sync::Arc;
use iqdb_index::{Index, IndexCore};
use iqdb_types::{DistanceMetric, SearchParams, VectorId};

// `FlatIndex` implements both traits (see examples/custom_index.rs).
let mut index = FlatIndex::new(3, DistanceMetric::Euclidean, FlatConfig)?;
index.insert(VectorId::from(1u64), Arc::from([1.0, 0.0, 0.0].as_slice()), None)?;
let hits = index.search(&[1.0, 0.0, 0.0], &SearchParams::new(1, DistanceMetric::Euclidean))?;
assert_eq!(hits[0].id, VectorId::U64(1));
# Ok::<(), iqdb_types::IqdbError>(())
```

## What's new in 0.2.0

### `IndexCore` — the object-safe operational surface

Every concrete index implements `IndexCore`: `insert`, `delete`, `search`, `len`, `dim`, `metric`, `flush`, and `stats`. The trait carries a `Send + Sync` bound so the engine can hold a heterogeneous set of indexes as `Box<dyn IndexCore>` and share them across threads. It is carefully kept object-safe — no generic methods, no `Self`-by-value returns — so `Box<dyn IndexCore>` compiles.

### `Index` — typed construction

A `Self`-returning constructor with an associated `Config` is *not* object-safe, so construction is split into a second trait. `Index: IndexCore` adds `type Config: Default + Clone` and `fn new(dim, metric, config) -> Result<Self>`. Each backend exposes its own parameter struct rather than a shared god-config enum; an index with nothing to tune uses a zero-sized unit config. The engine constructs through `Index::new`, then stores the result as `Box<dyn IndexCore>`.

### Default batch shims

`insert_batch` (fail-fast: the first error returns, prior inserts remain) and `search_batch` (order-preserving) ship with default implementations that loop over the single-item methods, alongside a default `is_empty`. A backend with a vectorized fast path overrides them; everyone else gets correct behaviour for free.

### `IndexStats` — uniform introspection

A snapshot of `n_vectors`, `memory_bytes`, an optional `disk_bytes` (`None` for in-memory indexes, not `0`), a stable `index_type` string, and an open-ended `extra` map for per-kind counters (tombstones, graph layers, training progress). `extra` is `Option<HashMap<…>>` with a `None` default, so the common `stats()` call on a dashboard allocates no empty map.

### Documented contracts

Three guarantees live on the trait, not on any one implementation:

- **Ordering** — `Hit.distance` is smaller-is-nearer and results come back best-first. `DotProduct` is a similarity (larger is closer), so an index must negate it at the boundary (`-dot`) to keep one ordering invariant across the family.
- **Deletion** — after `delete(id)`, `search` must not return `id` until a later `insert(id, …)` succeeds; whether storage is reclaimed, tombstoned, or compacted is implementation-defined and surfaced via `IndexStats::extra`.
- **Concurrency** — `IndexCore: Send + Sync`; an index need only be single-writer-internal, since the engine guards it with an external `RwLock`.

## Breaking changes

**Pre-1.0 API churn.** Relative to the v0.1.0 scaffold (no domain logic), v0.2.0 is the first real surface — everything in *What's new* is an addition. The crate is on a 0.x line; expect refinements as the real index consumers compile against it, until the API freezes at 1.0. The `std`/`serde` scaffold features were removed: the crate is std-only with no optional surface.
