# iqdb-index v0.3.0 — Validated Against the Real Indexes

**The design holds.** v0.3.0 puts the v0.2.0 trait surface to the test against the three index families it exists to unify — brute-force, graph, and clustered — and they fit it verbatim. No API change was required. This is the roadmap's "validate against `iqdb-flat`; refine the trait" phase, closed with a consumer-simulation suite and zero refinement.

## What is iqdb-index?

The polymorphism layer of the iQDB vector database: the `IndexCore` operational trait, the `Index` typed-construction trait, and `IndexStats`. The engine holds indexes as `Box<dyn IndexCore>` and runs the right strategy without being hardcoded to one. Its only dependency is `iqdb-types`.

## What's new in 0.3.0

### Cross-checked against the live consumers

The surface was validated against the real implementations in the sibling crates, each of which implements `IndexCore` + `Index` **verbatim** — same method signatures, its own associated `Config`, `extra: None` from `stats`, and an `is_empty` override:

| Crate | Type | `Config` | Family |
|---|---|---|---|
| `iqdb-flat` | `FlatIndex` | `FlatConfig` (unit) | brute-force |
| `iqdb-hnsw` | `HnswIndex` | `HnswConfig { m, ef_construction }` | graph |
| `iqdb-ivf` | `IvfIndex` | `IvfConfig { n_clusters, n_probes }` | clustered |

`FlatIndex` also confirmed the `Arc<[f32]>` insert contract end to end: it stores the caller's `Arc` without reallocating the payload (pointer-identity proven in its own test suite), which is exactly why the trait takes the vector by shared `Arc` rather than by slice.

### Consumer-simulation suite

`tests/consumer_simulation.rs` encodes a stand-in for each family at its exact construction shape and asserts:

- each constructs through `Index::new(dim, metric, config)` with its **own** `Config` (not a shared god-config enum), and rejects its own invalid knobs (`m == 0`, `n_clusters == 0`) plus the shared `dim == 0` check;
- each reports its own `index_type` (`"flat"` / `"hnsw"` / `"ivf"`) and `extra: None`;
- all three **coexist** in one `Vec<Box<dyn IndexCore>>` and are driven through `insert_batch` / `delete` / `search` / `flush` / `stats` with the concrete type erased.

This is DIRECTIVES §8 made concrete: the trait is implementable by graph, clustered, and brute-force indexes without awkward abstractions.

### On benchmarks

The roadmap's v0.3 exit criterion mentions benchmarking "where it is a hot path." `iqdb-index` is pure interface code — the only logic it ships is the default `insert_batch` / `search_batch` shims, which are O(n) wrappers that dispatch to the single-item methods. There is no hot path *in this crate* to benchmark; a consumer's search hot path is measured in that consumer's own crate (e.g. `iqdb-flat`'s `benches/flat_search.rs`). This is recorded honestly rather than padded with a meaningless micro-benchmark.

## Breaking changes

**None.** The public surface is unchanged from v0.2.0; this release is the validation milestone. The crate remains on a 0.x line until the API freeze.

## Verification

Run on Windows x86_64 (Rust stable 1.95.0). The same commands run in the CI matrix across Linux, macOS, and Windows at both `stable` and the `1.87` MSRV floor:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
```

All green. Test counts at this tag: **30 total** — 16 trait-shape + 5 property + 3 `IndexStats` + 3 consumer-simulation integration tests, plus 3 doctests. Zero `unsafe` (`#![forbid(unsafe_code)]`).
