# iqdb-index v1.0.0 — Stable

**The index contract is stable.** v1.0.0 commits the public API of `iqdb-index` under SemVer for the entire 1.x series: no breaking changes until 2.0. This is the seam every iQDB index plugs into — the trait the engine calls so flat, HNSW, and IVF are interchangeable — and it is now a fixed point the index crates and the database can build on without fear of churn. Nothing changed in the code since 0.5.0; this release *is* the stability commitment.

## What is iqdb-index?

The polymorphism layer of the iQDB vector database. Two traits and one struct:

- **`IndexCore`** — the object-safe operational surface (`insert`, `delete`, `search`, `len`, `dim`, `metric`, `flush`, `stats`, plus default `insert_batch` / `search_batch` / `is_empty` shims). The engine holds indexes as `Box<dyn IndexCore>`.
- **`Index`** — typed construction: an associated `Config: Default + Clone` and `new(dim, metric, config)`, split out so `IndexCore` stays object-safe.
- **`IndexStats`** — uniform, allocation-light runtime introspection.

Its only dependency is `iqdb-types`.

```rust
use std::sync::Arc;
use iqdb_index::{Index, IndexCore};
use iqdb_types::{DistanceMetric, SearchParams, VectorId};

// `FlatIndex` implements both traits (see examples/custom_index.rs).
let mut index = FlatIndex::new(3, DistanceMetric::Euclidean, FlatConfig)?;
index.insert(VectorId::from(1u64), Arc::from([1.0, 0.0, 0.0].as_slice()), None)?;
let hits = index.search(&[1.0, 0.0, 0.0], &SearchParams::new(1, DistanceMetric::Euclidean))?;
assert_eq!(hits[0].id, VectorId::U64(1));
# Ok::<(), iqdb_types::IqdbError>(())
```

## Why 1.0 now

The road here was deliberate, one verified phase at a time:

- **0.2.0 — The Index trait.** The object-safe `IndexCore` / typed `Index` split, `IndexStats`, and the default batch shims, with property tests for the core invariants (best-first ordering, deletion visibility, batch == loop).
- **0.3.0 — Validated against the real indexes.** Cross-checked against the live `iqdb-flat` (brute-force), `iqdb-hnsw` (graph), and `iqdb-ivf` (clustered) implementations — each implements the traits **verbatim** with its own `Config` — and a consumer-simulation proved all three coexist behind `Box<dyn IndexCore>` (DIRECTIVES §8). No trait change was needed.
- **0.4.0 — Async decision + feature freeze.** Recorded the synchronous-by-design decision (no async trait would survive object safety on the search hot path; the work is CPU-bound; async belongs at the engine boundary) and froze the feature set at empty.
- **0.5.0 — API freeze.** Locked the public surface and recorded it; documented each implementation's deletion semantics (flat/ivf true removal, hnsw tombstone — all honouring one observable contract).

By 0.5.0 every Definition-of-Done criterion (`dev/DIRECTIVES.md` §7) was met. The 0.6–0.9 RC band exists to soak the API against live consumers; in the build order those consumers come *after* this crate and do not yet exist as published crates, and the band's intent was already met by the consumer-simulation cross-checked against the real implementations. So the crate proceeds to 1.0 on a fully-satisfied checklist rather than a calendar — the same path `iqdb-types` took from 0.5 to 1.0.

## The 1.x compatibility promise

- The public surface recorded in `dev/ROADMAP.md` is frozen until 2.0.
- Additive, non-breaking changes remain allowed within 1.x: new **provided** trait methods (with defaults, so existing implementers are unaffected) and new public items.
- The `IndexCore` / `Index` split, the synchronous design, the `Arc<[f32]>` payload, and the `IndexStats` field set are part of the frozen contract.
- `IndexStats` is **not** `#[non_exhaustive]`: new per-kind detail goes in the `extra` map, never in a new field (which would be breaking).

## Breaking changes

**None.** No public API changed from 0.5.0; 1.0.0 is the SemVer stability commitment. v1.0.0 adds only an edge-case test suite.

## Verification

Verified on Windows x86_64 and Linux x86_64 (WSL2 Ubuntu), each on stable (1.95.0) and the 1.87 MSRV (`cargo +1.87`). The CI matrix runs the same across Linux, macOS, and Windows at stable and 1.87:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo deny check
cargo audit
```

All green on every platform/toolchain combination. **42 tests** — 16 trait-shape + 5 property + 3 `IndexStats` + 3 consumer-simulation + 12 edge-case integration tests, plus 3 doctests — alongside 4 runnable examples. **Zero `unsafe`** (`#![forbid(unsafe_code)]`). `cargo deny check` reports `advisories ok, bans ok, licenses ok, sources ok`; `cargo audit` finds zero advisories.

**Loom and benchmarks are deliberately N/A:** the crate is trait definitions plus one plain struct, with no concurrent or lock-free path and no hot path of its own. The `Send + Sync` bound is a compiler-checked requirement on implementers; a consumer's search hot path is benched in that consumer's crate.

## What's next

`iqdb-index` is done. The index crates build on it: `iqdb-flat`, `iqdb-hnsw`, and `iqdb-ivf` implement these traits; `iqdb-build`, `iqdb-eval`, and the `iqdb` database are generic over them. Anything those consumers surface that this crate should provide will be an additive `1.0.x` release.
