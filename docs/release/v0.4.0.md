# iqdb-index v0.4.0 — Sync by Design, Features Frozen

**Two commitments, locked.** v0.4.0 settles the question every trait-based library eventually faces — *async or not?* — and freezes the feature set. There is no code or public-surface change here; this is the decision milestone that clears the path to the API freeze. The answer to async is **no async trait**, recorded with its reasoning so it never has to be relitigated.

## What is iqdb-index?

The polymorphism layer of the iQDB vector database: the `IndexCore` operational trait, the `Index` typed-construction trait, and `IndexStats`. The engine holds indexes as `Box<dyn IndexCore>` and runs the right strategy without being hardcoded to one. Its only dependency is `iqdb-types`.

## What's new in 0.4.0

### Synchronous by design

The trait stays **synchronous** — `search`, `insert`, and the rest return `Result<T>`, not futures — and that is now a frozen decision, for three reasons:

1. **Object safety on the hot path.** `IndexCore` must be `dyn`-compatible so the engine can hold `Box<dyn IndexCore>`. An `async fn` in the trait is not `dyn`-compatible without boxing the returned future — a heap allocation on *every* `search` call, which the query hot path cannot afford.
2. **The work is CPU-bound.** A nearest-neighbour search is an in-memory scan or graph walk, not I/O. Wrapping CPU-bound work in a future buys nothing and costs a state machine.
3. **Async belongs at the engine boundary, not the index.** The engine already guards each index with an `RwLock`; if it wants an async API edge it offloads the blocking call (e.g. via `spawn_blocking`). That keeps the index contract simple and leaves the runtime choice to the engine.

Consequently there is **no async trait, no `async` feature, and no `futures` dependency**. Async is optional consumer-side *wrapping*, not part of this crate's surface. The rationale now lives in the crate-level docs and `docs/API.md` so it is discoverable and durable.

### Feature freeze

The feature set is **frozen at empty**:

- no `std` / `no_std` split — the trait uses `std::sync::Arc` and `std::collections::HashMap`, and its single dependency (`iqdb-types`) is `std`;
- no `serde` gate — the surface is *behaviour* (a trait), not data to serialize; `IndexStats` is a plain struct a consumer can serialize on its own side if needed;
- no `async` gate — see above.

Any feature introduced in the 1.x series would be purely additive, so this freeze costs consumers nothing and removes a configuration axis from the API-freeze review.

## Breaking changes

**None.** No code, no public surface, no dependency change from v0.3.0. This release records design decisions only.

## Verification

Run on Windows x86_64 (Rust stable 1.95.0). The same commands run in the CI matrix across Linux, macOS, and Windows at both `stable` and the `1.87` MSRV floor:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
```

All green. Test counts are unchanged at this tag: **30 total** — 16 trait-shape + 5 property + 3 `IndexStats` + 3 consumer-simulation integration tests, plus 3 doctests. Zero `unsafe` (`#![forbid(unsafe_code)]`); zero `todo!` / `unimplemented!`.

## What's next

- **v0.5.0 — per-impl deletion semantics + API freeze.** Record each implementation's deletion behaviour (true removal vs. tombstone) against the trait's deletion contract, then **freeze the public API** and record the frozen surface. After that the crate is on the 0.6–0.9 RC track, with 1.0 gated on integration against the live consumers.
